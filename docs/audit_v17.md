## Верификация исправлений audit_v16

### Fix A.2 — 🟢 PID в tmp файлах

Буфер `char tmp_v4[64]`, `tmp_v6[64]`, `tmp_fst[64]`.

Форматы:
- `/tmp/cdn_cf_v4_%d.txt` → 15 + 10 (max PID 4194304 = 7 цифр) + 4 + 1 = **30 байт**
- `/tmp/cdn_cf_v6_%d.txt` → аналогично 30 байт
- `/tmp/cdn_fastly_%d.json` → 16 + 7 + 5 + 1 = **29 байт**

Все влезают в 64 байта с запасом. `int pid = (int)getpid()` корректно, `pid_t` — знаковый 32-бит. На Linux max PID = 4194304, что даёт 7 цифр, не 10.

**Вердикт: исправлено верно.**

### Fix A.5 — 🟢 Strict aliasing в cmp_cidr_str

```c
return strcmp(*(const char (*)[64])a, *(const char (*)[64])b);
```

Правильное приведение через указатель на массив `[64]`. Избегает UB strict aliasing при разыменовании `const void *` как `char **`. Соответствует стандарту C, clang/gcc не выдают предупреждений.

**Вердикт: исправлено верно.**

### Fix A.8.3 — 🟢 Флаг write_err, ferror, одиночный fclose

Структура `cdn_merge_write` (строки 204–238):

```c
int write_err = 0;
if (fprintf(...) < 0) write_err = 1;

for (...) {
    if (fprintf(f, "%s\n", ...) < 0) { write_err = 1; break; }
}

if (!write_err && ferror(f)) write_err = 1;  // только если ещё не помечен
if (fclose(f) != 0)         write_err = 1;  // единственный fclose

if (write_err) { unlink(tmp_path); return -1; }
rename(tmp_path, out_path);
```

`fclose` вызывается ровно один раз. После него файл `f` не используется. Двойного `fclose` нет. `ferror` проверяется до `fclose` (корректный порядок). Если `write_err` уже установлен из `fprintf`, `ferror` не вызывается (оптимизация).

В `cdn_stamp_write` две ветки: либо `fclose(f)` при ошибке `fprintf` (строка 39), либо `fclose(f)` в конце (строка 43). Пути взаимно исключаются через `return -1`.

**Вердикт: исправлено верно. Двойного fclose нет.**

### Fix A.8.5 — 🟡 Валидация CIDR в extract_json_array — тихое отбрасывание

## Новые находки

### N.1 — 🔴 Отрицательный interval_days вызывает бесконечный цикл обновлений

**Файл:** `cdn_updater.c`, строки 66–82 и 376–397

`cdn_updater_check` проверяет только `interval_days == 0` (выключено). Отрицательные значения не перехватываются:

```c
// cdn_updater_check — строка 376:
if (cfg->cdn_update_interval_days == 0) { ... return 0; }
// Отрицательное значение проходит дальше

// cdn_is_stale — строка 66:
if (interval_days == 0) return -1;
// При interval_days = -1:
//   age_sec > (long)(-1) * 86400  →  age_sec > -86400
//   Всегда истинно (age_sec ≥ 0)  →  возвращает 1 (всегда устарел)
```

**Результат:** при `cdn_update_interval_days = -1` (случайная ошибка конфигурации или UCI) каждый вызов `cdn_updater_check` запускает `cdn_updater_update`, что приводит к:
- бесконечным HTTP-запросам к cloudflare.com и api.fastly.com каждые 30 секунд (интервал health-check);
- нагрузке на сеть и CPU;
- возможному rate limiting со стороны CDN-провайдеров.

**Исправление (не делать сейчас):** в `cdn_updater_check` добавить `if (cfg->cdn_update_interval_days < 0) return 0;`, или добавить проверку `interval_days > 0` в `cdn_is_stale`.

**Серьёзность: 🔴** — логическая ошибка, DoS-эффект при некорректном конфиге.

### N.2 — 🟡 Вводящий в заблуждение комментарий в cdn_updater_check

**Файл:** `cdn_updater.c`, строка 395

```c
if (stale <= 0) return 0;  /* актуально или выключено */
```

К этой строке `cdn_is_stale` **не может вернуть -1** (выключено), потому что `interval_days == 0` уже обработан выше (строка 376) с ранним `return 0`. Поэтому `stale` может быть только `0` или `1`. Комментарий «или выключено» — ложный вариант, дезориентирует при чтении кода.

**Серьёзность: 🟡** — только документационная неточность, логика правильная.

### N.3 — 🟡 Параметр cidr_size в публичном API потенциально опасен

**Файл:** `cdn_updater.c`/`cdn_updater.h`

Функции `cdn_parse_text` и `cdn_parse_fastly_json` принимают `int cidr_size`, но массив объявлен как `char cidrs[][64]` — фиксированный размер слота 64 байта.

Несоответствие:
- Если внешний вызывающий код передаст `cidr_size > 64`, `memcpy` может записать до `cidr_size - 1` байт в слот размером 64 → **переполнение буфера**.
- `qsort` в `cdn_merge_write` жёстко использует шаг `64`, игнорируя `cidr_size` (его нет в этой функции вовсе).

В **текущем** коде все внутренние вызовы используют `64`. Но публичное объявление в `.h` ничего не запрещает. Будущий вызывающий код с `cidr_size=128` приведёт к silent corruption.

**Серьёзность: 🟡** — API design hazard. В нынешнем коде не эксплуатируется.

### N.4 — 🟡 UB при экстремальном значении timestamp в stamp-файле

**Файл:** `cdn_updater.c`, строки 67–82

```c
long ts = cdn_stamp_read(stamp_path);  // fscanf("%ld", &ts) — без ограничения диапазона
if (ts < 0) return 1;                  // перехватывает ts < 0, включая LONG_MIN
long now = (long)time(NULL);
long age_sec = now - ts;               // UB если ts = LONG_MAX и now ≈ 1.7e9
```

Если stamp-файл содержит `LONG_MAX` (9223372036854775807), то:
- `ts ≥ 0` → не перехвачен строкой 68
- `age_sec = 1776000000 - 9223372036854775807` → **знаковое переполнение = UB по стандарту C**

На практике (x86, two's complement): результат — большое отрицательное число → `age_sec < 0` → `return 1` (stale). Поведение «безопасное» но формально UB.

Stamp-файл записывается в `/etc/4eburnet/dpi/ipset.stamp` — только root имеет доступ на запись. Вектор атаки требует root-привилегий.

**Серьёзность: 🟡** — UB, но не эксплуатируется без root. На практике не проявляется.

### N.5 — 🟢 Второй вызов extract_json_array — арифметика смещения верна

**Файл:** `cdn_updater.c`, строки 171–175

```c
int n = 0;
n += extract_json_array(json, "addresses",      cidrs, n, max_count, cidr_size);
n += extract_json_array(json, "ipv6_addresses", cidrs, n, max_count, cidr_size);
```

Внутри `extract_json_array`: `int n = start`, пишет в `cidrs[n]` при условии `n < max_count`, возвращает `n - start` (кол-во добавленных).

Сценарий: первый вызов записывает `k` записей в `cidrs[0..k-1]`, возвращает `k`. Внешний `n = k`. Второй вызов: `start = k`, пишет в `cidrs[k..]`, ограничен `max_count`. Возвращает добавленное. Суммарно не превышает `max_count`.

**Вердикт: корректно.**

## Резюме

Все исправления из audit_v16 применены корректно. Критических ошибок, внесённых при правке, нет.

Единственная новая **красная** находка (N.1): отрицательное значение `cdn_update_interval_days` не перехватывается, что приводит к тому, что `cdn_is_stale` возвращает `1` при каждом вызове и `cdn_updater_update` запускается на каждом тике event loop. Исправляется одной строкой guard-а в `cdn_updater_check` или в `cdn_is_stale`.

Жёлтые находки N.2–N.4 не требуют срочного исправления: N.2 — документационная неточность, N.3 — API-предупреждение без текущего вектора, N.4 — UB только с root-записью LONG_MAX в stamp.
