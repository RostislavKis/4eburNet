# Devil Audit v16 — cdn_updater + config CDN ветви

**Дата:** 2026-04-11
**Файлы:** `cdn_updater.h`, `cdn_updater.c`, `test_cdn_updater.c`, `config.c` (CDN ветви)

## A. cdn_updater.c

### A.1 — 🟡 fetch_and_parse: ferror() не проверяется, ошибка чтения неотличима от пустого файла

`fread()` возвращает 0 как при EOF, так и при ошибке чтения. После вызова
`fread(buf, 1, CDN_MAX_DOWNLOAD_BYTES - 1, f)` код не проверяет `ferror(f)`.
Если файл был успешно создан `net_http_fetch()` но fread упал (например,
прерван сигналом, EINTR на медленном POSIX-IO), `total == 0`, `buf[0] == '\0'`.
Парсер вернёт 0, источник будет молча проигнорирован, как будто скачан пустой
ответ. Реальная ошибка не логируется. При нулях со всех трёх источников
`cdn_updater_update` вернёт `-1` без объяснения причины.

**Строки:** 225–227 (`fread` + `buf[total] = '\0'`).

**Рекомендация (не исправлять):** добавить `if (ferror(f)) log_msg(LOG_WARN, ...)`
между `fread` и `fclose`.

---

### A.2 — 🔴 Гонка по /tmp/*.txt: параллельный вызов cdn_updater_update затирает файлы

Временные файлы захардкожены: `/tmp/cdn_cf_v4.txt`, `/tmp/cdn_cf_v6.txt`,
`/tmp/cdn_fastly.json`. Если `cdn_updater_update` вызван дважды одновременно
(например, IPC «update-ipset» во время планового вызова из event loop — даже
при однопоточной архитектуре это возможно если используются fork/thread в
`net_http_fetch`), оба пишут в один и тот же файл, и второй вызов прочитает
мусор или частичные данные первого.

Архитектура задекларирована как однопоточная, но:
1. В заголовке `cdn_updater.h` нет ни одного комментария про non-reentrant.
2. `net_http_fetch` может внутри себя fork'ать wget/curl (типично для OpenWrt).
3. Тесты не проверяют поведение при конкурентном вызове.

Дополнительно: имена `/tmp/cdn_*.txt` конфликтуют с другими процессами в
системе (другой экземпляр демона, ручной скрипт).

**Строки:** 270–283 (три вызова `fetch_and_parse` с фиксированными путями).

**Рекомендация (не исправлять):** использовать `mkstemp()` или добавить PID
в имя файла. Добавить флаг «обновление уже идёт» (atomic flag).

---

### A.3 — 🟢 calloc(2048, 64) = 128 KB: OOM проверен, всё корректно

`calloc` возвращает `NULL` при нехватке памяти — проверка присутствует
(строка 261–264). На MICRO-профиле (64 MB RAM) 128 KB — приемлемо.
`calloc` обнуляет память, поэтому нет риска чтения неинициализированных
байт в `cidrs[n][0]`. Нарушений не найдено.

---

### A.4 — 🟢 Арифметика total: переполнение 2048 невозможно

Каждый `fetch_and_parse` получает `max = CDN_MAX_CIDRS_TOTAL - total` и
возвращает `n ≤ max`. После каждого вызова `total += n`, при этом
`total ≤ CDN_MAX_CIDRS_TOTAL`. Инвариант сохраняется при любом числе
источников. Выход за пределы массива `all[CDN_MAX_CIDRS_TOTAL]` невозможен.
Нарушений не найдено.

---

### A.5 — 🔴 cmp_cidr_str: неверный cast — qsort передаёт указатели на элементы, а не char*

```c
static int cmp_cidr_str(const void *a, const void *b)
{
    return strcmp((const char *)a, (const char *)b);
}
```

Массив объявлен как `char (*all)[64]`, элемент — `char[64]`.
`qsort` передаёт в компаратор **указатели на элементы**, то есть
`a` и `b` имеют тип `char (*)[64]` приведённый к `void *`.
Приведение `(const char *)a` даёт `const char *`, указывающий на первый
байт первого элемента пары — это **случайно правильно** для плоского
`char[64]` массива, потому что `&arr[i]` == `&arr[i][0]` в памяти.

Однако формально это нарушение strict aliasing и UB согласно C23:
указатель типа `char (*)[64]` нельзя разыменовывать как `char *` без
явного reinterpret через `memcpy` или подобного. На практике на
всех целевых архитектурах (x86\_64, mipsel, aarch64, armv7) это работает,
но компилятор с `-fstrict-aliasing -O2` теоретически вправе нарушить
ожидаемый порядок.

Правильный вариант:
```c
return strcmp(*(const char (*)[64])a, *(const char (*)[64])b);
// или:
return strcmp((const char *)a, (const char *)b);  // работает, но UB по стандарту
```

Реально достаточно `return strcmp(a, b)` — implicit decay массива в указатель.

**Строки:** 159–162 + вызов `qsort` на строке 169.

### A.6 — 🟡 cdn_stamp_write: при ошибке fprintf stamp остаётся частично записанным, unlink не вызывается

```c
if (fprintf(f, "%ld\n", (long)time(NULL)) < 0) {
    fclose(f);
    return -1;   // ← файл НЕ удаляется
}
```

Если `fprintf` упал после частичной записи (редко, но возможно при
переполнении диска в /tmp), файл содержит неполное число. `fscanf(f, "%ld")`
в `cdn_stamp_read` вернёт -1 (или 0 байт), `cdn_is_stale` вернёт 1 (stale).
Результат: лишняя попытка обновления при следующем старте — не катастрофа,
но поведение немного неожиданное. Аналогично если `fclose` вернул ошибку
(строка 44–48) — в этом случае данные не сброшены, а функция вернула `-1`,
но файл остался.

Более серьёзный случай: stamp лежит в `/etc/4eburnet/dpi/` (не tmpfs),
переполнение flash влечёт именно этот сценарий.

**Строки:** 38–48.

### A.7 — 🟡 cdn_is_stale: будущий timestamp (NTP не синхронизирован) → никогда не обновляется

```c
long age_sec = (long)time(NULL) - ts;
return (age_sec > (long)interval_days * 86400) ? 1 : 0;
```

Если системное время при записи stamp было в будущем (NTP ещё не
работал, часы показывали +N дней), а потом NTP откатил время назад,
`age_sec` становится **отрицательным**. Условие `age_sec > N * 86400`
ложно → `return 0` («свежий»). Демон не будет обновлять ipset до тех
пор, пока реальное время не догонит timestamp в stamp файле.

На практике архитектура предусматривает NTP Bootstrap до старта wolfSSL
(DEC-019), но `cdn_updater_check` вызывается **до** полной инициализации
сети. Если NTP Bootstrap не завершился и stamp уже есть — проблема реальна.

Верхняя граница: если stamp = now+7d, «заморозка» = 7 суток. Молча.

**Строки:** 69–71.

### A.8 — Прочие находки

#### A.8.1 — 🟡 fetch_and_parse: malloc(512KB) на стеке — утечка при возврате из-за OOM в parse

```c
char *buf = malloc(CDN_MAX_DOWNLOAD_BYTES);  // 512 KB
...
int n = is_json ? cdn_parse_fastly_json(buf, ...) : cdn_parse_text(buf, ...);
free(buf);
return (n > 0) ? n : 0;
```

Если парсер вернёт `-1` (ошибка параметров — но она невозможна при
корректных аргументах), `free(buf)` всё равно вызывается. Утечек нет.
Однако 512 KB на один fetch × 3 источника = 1.5 MB в пике (последовательно,
не одновременно). На MICRO-профиле (64 MB) — терпимо, но malloc+fread
по 512KB три раза подряд без jemalloc/musl может фрагментировать heap.
Не баг, но наблюдение.

#### A.8.2 — 🔴 cdn_updater_update: cdn_stamp_write вызывается без проверки возвращаемого значения

```c
cdn_stamp_write(stamp_path);   // строка 305, возвращаемое значение игнорируется
```

Если stamp не записался (например, /etc/4eburnet/dpi/ read-only),
следующий вызов `cdn_updater_check` снова найдёт stamp отсутствующим,
снова вызовет `cdn_updater_update` — лишние скачивания при каждом старте.
Ошибка залогирована внутри `cdn_stamp_write`, но вызывающий код не знает
о проблеме и возвращает `0` (успех), хотя состояние не персистировано.

**Строки:** 305.

#### A.8.3 — 🟡 cdn_merge_write: fprintf для строк CIDR не проверяет ошибки

```c
fprintf(f, "%s\n", cidrs[i]);
written++;
```

Ошибки записи не проверяются. Если диск/tmpfs заполнен в середине
записи, функция продолжает итерацию, затем делает `rename()`, и
итоговый файл окажется **неполным** без какого-либо сигнала вызывающему.
`fclose(f)` тоже не проверяется перед `rename`.

**Строки:** 190–192, 193.

#### A.8.4 — 🟢 cdn_merge_write: tmp_path size 544 — корректен

`out_path` max = 511 (snprintf с sizeof=512), суффикс `.tmp` = 4 байта,
NUL = 1. Итого 516. `tmp_path[544]` — запас достаточный. Переполнения нет.

#### A.8.5 — 🟡 extract_json_array: нет защиты от JSON с escape-последовательностями внутри строк

Ручной парсер ищет `"` как конец значения. Если CIDR-строка содержит
`\"` (валидный JSON escape), парсер обрежет строку на backslash. В
реальных ответах Fastly IP-адреса экранирования не содержат, но API
может добавить метаданные или поменять формат. Без валидации CIDR после
парсинга в ipset.txt могут попасть мусорные строки.

**Строки:** 126–141.

## B. config.c — CDN ветви

### B.1 — 🟡 cdn_cf_v4_url: минимальная валидация — "https://" без хоста принимается

```c
if (strncmp(value, "https://", 8) != 0)
    log_msg(LOG_WARN, ...);
else
    snprintf(cfg->cdn_cf_v4_url, ...);
```

Значение `https://` (без хоста) проходит проверку и записывается в
конфиг. `net_http_fetch("https://", ...)` вернёт ошибку — источник
будет молча пропущен. Поведение предсказуемое (graceful degrade), но
опечатка в конфиге не диагностируется. Аналогично для cdn_cf_v6_url и
cdn_fastly_url.

**Строки:** 201–227.

---

### B.2 — 🟢 cdn_update_interval_days "7.5" → не применяется, warn — корректно

`strtol` при "7.5" возвращает 7, `*ep == '.'` ≠ `'\0'` → LOG_WARN,
значение не меняется, остаётся default. Поведение правильное.

---

### B.3 — 🟡 cdn_update_interval_days=0: первый старт без ipset.txt — dpi_filter загружает пустой список, не задокументировано

При `interval_days=0` обновление выключено. Если ipset.txt не существует
(первый старт, flash чистый), `cdn_updater_check` возвращает `0` без
создания файла, а `dpi_filter_init` загружает пустой или несуществующий
ipset. Трафик через DPI не маршрутизируется на CDN. Это молчаливое
отключение функциональности без предупреждения в логах.

Документация (CLAUDE.md, README) не описывает это поведение. Оператор,
выставив `0` «для отключения авто-обновлений», может рассчитывать на
ручной запуск через IPC «update-ipset», но не осознавать, что при чистом
старте ipset пуст.

**Строки:** config.c:318 (`cdn_updater_check` ранний return 0).

## C. test_cdn_updater.c

### C.1 — 🟡 test_updater_no_network: HTTPS на 127.0.0.1:1 — поведение зависит от net_http_fetch реализации

URL `https://127.0.0.1:1/ips-v4` предполагает ECONNREFUSED (порт 1 закрыт).
На Linux/WSL это действительно быстро (< 1ms). Однако:

1. Если `net_http_fetch` внутри вызывает `curl` или `wget` с TLS, до
   TCP connect не дойдёт — TLS handshake не начнётся, ECONNREFUSED
   сразу. Хорошо.
2. Если `net_http_fetch` использует DNS (маловероятно для IP), никаких
   задержек нет.
3. **Проблема**: если `net_http_fetch` устанавливает таймаут соединения
   отдельно от таймаута ответа (например, `--connect-timeout 30`),
   то на WSL с включённым Windows Firewall порт 1 может не возвращать
   RST, а дропать пакеты → таймаут = 30 секунд × 3 источника = 90 сек.
   Тест становится медленным.

Комментарий в коде «немедленный ECONNREFUSED, без сети» — предположение,
не гарантия.

### C.2 — Прочие пробелы покрытия

#### C.2.1 — 🔴 Нет теста: cdn_is_stale с будущим timestamp (A.7)

Сценарий «stamp = now + 7 дней, age_sec < 0 → return 0 ("свежий")»
не протестирован. Добавление 8 строк кода обнаружило бы баг немедленно.

#### C.2.2 — 🟡 Нет теста: cdn_parse_text с CIDR длиной ровно 63 символа (граничное условие cidr_size)

`if (len > 0 && len < cidr_size)` — строка длиной `cidr_size-1` = 63
байта принимается, длиной 64 — нет. Граница не протестирована. IPv6
CIDR типа `2001:0db8:85a3:0000:0000:8a2e:0370:7334/128` — 43 символа,
в реальности в предел не упирается, но тест полезен.

#### C.2.3 — 🟡 Нет теста: cdn_parse_fastly_json с escape в JSON строках (A.8.5)

Fastly может вернуть пробел или спецсимвол. Парсер не тестируется на
edge cases JSON формата.

#### C.2.4 — 🟡 Нет теста: cdn_merge_write с count=0

Функция проверяет `count <= 0` и возвращает `-1`. Это не тестируется.
По контракту cdn_updater_update никогда не вызывает merge_write с 0,
но функция экспортирована — контракт должен быть покрыт тестом.

#### C.2.5 — 🟡 Нет теста: cdn_stamp_write при неписаемом пути

Сценарий «stamp_path в read-only директории → возврат -1» не
протестирован. Актуально для обнаружения A.8.2.

#### C.2.6 — 🟢 system() в setup/cleanup — приемлемо для тестов

`system("mkdir -p ...")` и `system("rm -rf ...")` в тестовом коде
допустимы. Путь `TMP_DIR` фиксирован и не содержит пользовательского
ввода. Инъекция не возможна.
