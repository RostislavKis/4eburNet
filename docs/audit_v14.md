## 🔴 Критично

### v14-1: `match_ipv4`/`match_ipv6` — произвольное окно `hi - 4` может пропустить совпадение

**Файл:** `core/src/dpi/dpi_filter.c`, функции `dpi_filter_match_ipv4` и `dpi_filter_match_ipv6`

```c
for (int i = hi; i >= 0 && i >= hi - 4; i--) {
    if (g_ipv4[i].addr > ip) continue;
    if ((ip & g_ipv4[i].mask) == g_ipv4[i].addr)
        return DPI_MATCH_BYPASS;
}
```

**Проблема:** После бинарного поиска `hi` = последний индекс где `addr <= ip`. Цикл проверяет
только `i = hi, hi-1, hi-2, hi-3, hi-4` — 5 элементов. Если между широким охватывающим
диапазоном и целевым IP вставлено 5+ более мелких диапазонов от других CDN, охватывающий
диапазон оказывается за границей окна и пропускается → false negative.

**Пример:**
```
ranges[0]: 10.0.0.0/8    addr=0x0A000000  ← охватывает 10.x.x.x
ranges[1]: 10.1.0.0/24   addr=0x0A010000
ranges[2]: 10.2.0.0/24   addr=0x0A020000
ranges[3]: 10.3.0.0/24   addr=0x0A030000
ranges[4]: 10.4.0.0/24   addr=0x0A040000
ranges[5]: 10.5.0.0/24   addr=0x0A050000
ip = 10.6.0.1 = 0x0A060001  ← должен дать BYPASS (в /8)

bsearch: hi = 5
цикл: i=5,4,3,2,1 → ни один не матчит
ranges[0] (10.0.0.0/8) ПРОПУЩЕН → DPI_MATCH_NONE  ← НЕВЕРНО
```

**Статус с текущим `ipset.txt`:** 0 случаев false negative (подтверждено анализом 4794 IPv4
CIDR — нет диапазонов с 5+ вложенными записями). Однако:
- Окно `hi - 4` не имеет обоснования в коде
- Запрет обновляет `ipset.txt` регулярно — при добавлении новых CDN блоков баг активируется
- `match_ipv6` содержит идентичный баг

**Исправление:**
```c
/* Убрать ограничение hi - 4, проверять всех кандидатов где addr <= ip */
for (int i = hi; i >= 0; i--) {
    if (g_ipv4[i].addr > ip) continue;
    if ((ip & g_ipv4[i].mask) == g_ipv4[i].addr)
        return DPI_MATCH_BYPASS;
}
```
Производительность: O(log N + K) где K — количество диапазонов с addr ≤ ip, на практике 1–3.

### v14-2: `atoi(slash + 1)` без валидации — plen=0 даёт маску 0 → матчит ВСЁ

**Файл:** `core/src/dpi/dpi_filter.c`, функция `parse_ipset`

```c
plen = atoi(slash + 1);
// ...
int pl = (plen >= 0 && plen <= 32) ? plen : 32;
uint32_t mask = prefix_to_mask4(pl);
```

**Проблема:**
- `atoi("")` = 0 (строка вида `"1.2.3.0/"` — слэш в конце строки, после stripping)
- `atoi("0abc")` = 0 (нечисловой суффикс)
- `plen = 0`, `pl = 0` (условие `0 >= 0 && 0 <= 32` = true)
- `prefix_to_mask4(0)`: `if (plen <= 0) return 0;` → `mask = 0`
- `addr & 0 = 0`, запись `{addr=0, mask=0}` добавляется в массив
- Матчинг: `(ip & 0) == 0` → `0 == 0` → **TRUE для ЛЮБОГО IP**

Результат: **каждый IP в ipset считается CDN** — весь трафик помечается как DPI_BYPASS.
Один malformed line в `ipset.txt` делает фильтр бесполезным (false positive по всему трафику).

**Исправление:** изменить условие `plen >= 0` на `plen > 0`:
```c
int pl = (plen > 0 && plen <= 32) ? plen : 32;
```
Тогда `plen = 0` → `pl = 32` (host route), что корректно для единичного хоста.
Аналогично для IPv6: `plen > 0 && plen <= 128`.

## 🟢 Улучшения

### v14-4: `parse_ipset` — два прохода по файлу, теоретический race condition

**Файл:** `core/src/dpi/dpi_filter.c`, функция `parse_ipset`

Первый проход подсчитывает строки (`n4`, `n6`), второй парсит. Если файл изменяется
между проходами (например, `opkg update` заменяет ipset.txt), счётчики и реальное
содержимое могут не совпасть. На OpenWrt при нормальной работе невозможно, но архитектурно
ненадёжно.

Альтернатива: динамический массив с `realloc` за один проход.

### v14-5: `parse_ipset` — `g_ipv4` не освобождается явно при OOM на `g_ipv6`

**Файл:** `core/src/dpi/dpi_filter.c`, функция `parse_ipset`

```c
g_ipv4 = calloc((size_t)(n4 + 1), sizeof(ipv4_cidr_t));  // успех
g_ipv6 = calloc((size_t)(n6 + 1), sizeof(ipv6_cidr_t));  // OOM → NULL
if (!g_ipv4 || !g_ipv6) {
    fclose(f);
    return -1;     // ← g_ipv4 не освобождён явно
}
```

`g_ipv4` (глобальный) указывает на валидную память, `g_ipv4_n = 0`.
`dpi_filter_init` продолжает без ipset (логирует warning). При последующем
`dpi_filter_free()` память корректно освобождается через `free(g_ipv4)`.

Утечки нет, но состояние неочевидно: `g_ipv4 != NULL` при `g_ipv4_n == 0`.
Рекомендация: явный `free(g_ipv4); g_ipv4 = NULL;` перед `return -1`.

### v14-6: `dpi_filter_init` не thread-safe

**Файл:** `core/src/dpi/dpi_filter.c`, функция `dpi_filter_init`

`dpi_filter_free()` в начале `init` и последующая загрузка не защищены mutex.
Параллельный вызов (например, из будущего reload по SIGHUP) приведёт к race condition.
Текущая однопоточная архитектура исключает проблему, но стоит задокументировать
как `/* NOT thread-safe: вызывать только из главного потока */`.
