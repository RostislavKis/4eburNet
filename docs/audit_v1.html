Топ-5 критичных находок

CRITICAL C-03 crypto/tls.c:190-214

Блокирующий TLS handshake внутри event loop.

tls_connect() содержит цикл select() с 50 итерациями × 100мс = до 5 секунд блокировки. Вызывается из dispatcher_tick() → vless_protocol_connect(). Пока идёт TLS handshake к одному серверу, ВСЕ остальные соединения заморожены: нет accept, нет relay, нет IPC. При медленном upstream или таймауте — весь прокси не работает 5 секунд.

Вынести TLS handshake в неблокирующий конечный автомат, интегрированный в epoll. wolfSSL поддерживает WOLFSSL_CBIO_RECV/WOLFSSL_CBIO_SEND для неблокирующего IO.
CRITICAL C-05 proxy/dispatcher.c:401-419

Общий splice pipe — повреждение данных между соединениями.

Один splice_pipe[2] используется всеми relay-соединениями. При splice из client_fd в pipe, если запись из pipe в upstream_fd прерывается (EAGAIN, partial write), данные остаются в pipe. Следующий вызов relay_transfer() для другого соединения прочитает чужие данные из pipe и отправит их не тому upstream. Это повреждение данных и утечка информации между соединениями клиентов.

Либо pipe-per-connection (дорого по fd), либо отказ от splice при наличии TLS, либо гарантированный drain pipe до конца перед возвратом.
CRITICAL C-06 main.c:362-368 + crypto/tls.c:148

Use-after-free при перезагрузке конфига.

При SIGHUP/reload: config_free(&cfg) освобождает cfg.servers. Но активные TLS-соединения в dispatcher хранят tls_conn.config.sni = server->address (shallow copy указателя в tls_connect(), строка 148). После free, все активные TLS relay имеют dangling pointer на SNI строку. wolfSSL может обратиться к освобождённой памяти при renegotiation или shutdown.

Копировать SNI строку через strdup() в tls_connect(), освобождать в tls_close(). Или не освобождать старый конфиг пока есть активные соединения.
CRITICAL C-08 main.c (отсутствует)

Нет обработки SIGPIPE — демон будет убит при write() в закрытый сокет.

При write() в сокет, чей peer уже закрылся, ядро отправляет SIGPIPE. Действие по умолчанию — завершение процесса. В сетевом демоне с множеством relay-соединений это неизбежный сценарий. Один клиент закрыл соединение неожиданно → весь демон падает.

Добавить signal(SIGPIPE, SIG_IGN) в начало main(), до любых сетевых операций.
CRITICAL C-04 proxy/protocols/vless.c:133-158

Блокирующий select() в vless_read_response() внутри event loop.

Аналогично C-03: цикл с select() до 5 секунд при чтении VLESS response. Вызывается из dispatcher_tick() → vless_handshake(). Усугубляет C-03: суммарная блокировка до 10 секунд (5 сек TLS + 5 сек VLESS).

Объединить с TLS handshake в общий неблокирующий конечный автомат.

core/src/config.c (264 строки)

Роль: Парсинг UCI-подобного конфига.

Находки:

  • H-03 Строка 102: atoi() для порта без валидации
  • Строка 125-126: strcpy() для default значений — безопасно (длины проверены), но стилистически лучше strncpy.

core/src/ntp_bootstrap.c (199 строк)

Роль: Установка времени через HTTP Date: заголовок при холодном старте.

Находки:

  • H-08 Захардкоженные IP-адреса bootstrap хостов (строки 27-34)
  • M-04 _XOPEN_SOURCE после includes (строка 15)
  • Строка 100: connect() блокирующий с SO_RCVTIMEO=3 сек. До 3 хостов × 3 сек = 9 секунд блокировки при старте. Допустимо при инициализации.
  • Строка 110: send(fd, req, req_len, 0) != req_len — partial send не обрабатывается. На blocking socket с маленьким запросом (~50 байт) это безопасно.
  • Строка 135-137: Поиск "Date: " case-insensitive только для двух вариантов. "DATE: " или "dAtE: " не найдутся. RFC допускает любой регистр.

Безопасность: settimeofday() требует CAP_SYS_TIME. На OpenWrt демон работает от root — OK. Но время от HTTP не аутентифицировано: MITM может подставить неверное время, вызвав принятие просроченных TLS сертификатов.

Архитектурные проблемы

ARCH-01: Блокирующие вызовы в event loop

Серьёзность: CRITICAL

Главная архитектурная проблема проекта. Event loop (main.c:350-378) должен быть полностью неблокирующим, но содержит:

  • tls_connect() — select() до 5 сек (crypto/tls.c:190-214)
  • vless_read_response() — select() до 5 сек (vless.c:133-158)
  • ipc_process() → read() — блокирующий read на client_fd (ipc.c:82)
  • ntp_bootstrap() — до 9 сек при старте (допустимо, одноразово)

При множественных соединениях к медленным upstream, демон фактически последовательно подключается к каждому, замораживая всю обработку. Один медленный сервер убивает весь прокси.

Хронология при 3 соединениях к медленному upstream:
  [0s]     conn1: tls_connect() начат
  [0-5s]   ВСЁ заморожено (tproxy, relay, IPC)
  [5s]     conn1: vless_handshake() начат
  [5-10s]  ВСЁ заморожено
  [10s]    conn2: tls_connect() начат
  [10-15s] ВСЁ заморожено
  ...итого 30 секунд недоступности

ARCH-02: Poll-sleep вместо event-driven

Серьёзность: HIGH

Два epoll (tproxy + dispatcher) поллятся с timeout=0, затем usleep(10000). Это добавляет:

  • 10мс latency floor для каждого события
  • 100 пробуждений/сек в idle (лишний CPU на роутере)
  • 2 лишних syscall (epoll_wait) на каждую итерацию

Правильный дизайн: один epoll_wait с timeout=10мс (или -1), включающий все fd из обоих модулей + IPC fd.

ARCH-03: Общий splice pipe

Серьёзность: CRITICAL

Один pipe[2] для splice() разделяется между всеми relay соединениями (dispatcher.c:59). При partial splice данные из pipe могут попасть в чужое соединение. Это:

  • Повреждение данных
  • Утечка данных между клиентами (privacy violation)
  • Непредсказуемое поведение при высокой нагрузке

Спасает то, что splice используется только когда !r->use_tls (direct протокол). Для VLESS (use_tls=true) splice не задействован. Но для future direct-протокола или debug-режима — бомба замедленного действия.

ARCH-04: WOLFSSL_CTX per connection

Серьёзность: HIGH

Каждое TLS соединение создаёт свой WOLFSSL_CTX (tls.c:153). CTX создание включает:

  • Аллокация памяти для cipher state
  • Парсинг cipher list строки
  • Инициализация RNG

Должен быть один CTX на fingerprint профиль (3 штуки), переиспользуемый для всех соединений.

ARCH-05: Дублирование sets и verdict maps

Серьёзность: MEDIUM

nft_init() создаёт обычные sets (bypass_addrs, proxy_addrs, block_addrs), а nft_vmap_create() создаёт verdict maps (bypass_map, block_map). Правила в цепочках ссылаются на обычные sets. Verdict maps создаются, но не используются в цепочках — правила используют @bypass_addrs, не vmap @bypass_map.

DEC-017 (Verdict Maps) объявлен реализованным, но фактически maps создаются и заполняются, но не задействованы в правилах фильтрации. Обычные sets по-прежнему используются.
